Agent eval is two-dimensional:

- **Trajectory.** What did the agent do? Did it use the right tools, in the right order, with the right arguments?
- **Outcome.** What did the agent achieve? Was the final state correct?

A correct outcome with a wrong trajectory is suspicious. A correct trajectory with a wrong outcome means the trajectory wasn't actually correct. Both axes need eval.

## Trajectory eval

For each case:

- Expected trajectory (tools called, in order).
- Actual trajectory (from agent traces).
- Comparison.

Strict comparison: tools must match.

Lenient comparison: equivalent tool patterns acceptable.

The right strictness depends on the use case.

## Outcome eval

For each case:

- Expected end state.
- Actual end state.
- Comparison.

End state might be: a created record, a sent message, a final response, a database state.

## Combined

Combined evaluation:

- Trajectory + outcome both correct: PASS.
- Trajectory wrong but outcome right: investigate (lucky correctness?).
- Trajectory right but outcome wrong: investigate (system issue downstream?).
- Both wrong: clear fail.

## Reviewer ritual

PR review:

- Both axes evaluated.
- Discrepancies investigated.
- Cohort patterns surfaced.

## A real implementation

A team's customer-research agent:

- 40 cases with trajectory annotations.
- 40 cases with outcome annotations.
- Both axes pass at 95%+ for merge.

Catch rate compared to outcome-only: 30% more issues found.

## Trade-offs

Trajectory annotation is harder than outcome annotation. The investment is in the eval-set authoring. Once authored, the eval is reliable.

## What we won't ship

**Agent eval** with only outcome scoring.

**Trajectory eval** that's so strict the agent has no flexibility.

**Skipping the discrepancy investigation.**

**Cohort blindness** to pattern-level issues.

## Close

Evals for agents need both axes. Trajectory captures how. Outcome captures what. Together, they tell the team whether the agent worked the way intended. Skip either and the eval has blind spots.
