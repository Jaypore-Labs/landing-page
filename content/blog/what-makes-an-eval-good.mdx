A good eval has three legs. Without any one, the eval misleads.

- **Observability.** You can see what passed and what failed.
- **Deciding power.** Pass or fail informs a real decision.
- **Repeatability.** Same eval, same code, same model gives same result.

If you have observability without deciding power, you have a dashboard. If you have deciding power without observability, you have a black-box gate. If you have either without repeatability, you have a vibe.

## Observability

The eval's results are readable:

- Per-case pass/fail.
- Failure reason for each fail.
- Trend over time.
- Per-category breakdown.

Without this, the team can't act on the eval. They see "the eval failed" and don't know why.

## Deciding power

The eval informs decisions:

- "Ship this PR" or "block it."
- "Update the prompt" or "leave it."
- "Investigate this drift" or "ignore it."

Without deciding power, the eval is theatre. Teams run it for show.

## Repeatability

The eval gives the same answer twice:

- Same inputs.
- Same outputs (within tolerance).
- Same scoring.

Without repeatability, the team can't trust the result. "It failed today but passed yesterday on the same code" — that's noise, not signal.

## A real eval reviewed

A team's classification eval:

- 200 cases. ✓ observability.
- Pass rate >95% required for merge. ✓ deciding power.
- Temp 0; same model version pinned; same scoring code. ✓ repeatability.

This eval drives engineering. Decisions are evidence-based. Drift catches issues early.
