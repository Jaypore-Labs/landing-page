# End-to-end content pipeline

> Article → site → X thread → Instagram carousel.
> A pragmatic plan, three phases, honest about what each one costs.

## What's already in place

- **Site publishing.** `git push origin main` → Vercel builds and deploys → article is live within ~3 minutes. No work to add here.
- **SEO ping.** `.github/workflows/indexnow.yml` notifies Bing/Yandex on every push.
- **Per-article OG + Twitter share images.** Auto-generated by `src/app/blog/[slug]/opengraph-image.tsx` and `twitter-image.tsx`.
- **Article generator.** `/daily-article` skill produces the MDX, IG carousel slide PNGs, IG caption, LinkedIn post, and X thread.
- **Image generator.** `scripts/gen-post.mjs` renders any social size on-brand.

So the *generation* side is done. What's missing is the *posting* side for X and Instagram. That's what this doc plans for.

---

## Three phases

We do this in three phases. Each phase ships value on its own. You can stop after any phase.

### Phase 1 — Manual posting (this week)

Generation is automated. Posting is manual. ~10 minutes per article post.

**Flow per article:**

1. Run `/daily-article <slug>`. Skill produces:
   - `content/blog/<slug>.mdx` (the article)
   - `social/carousels/<slug>/slide-*.png` (6 portrait IG slides)
   - `social/carousels/<slug>/caption.md` (IG caption + hashtags)
   - `social/linkedin-posts/<slug>.md` (LinkedIn post)
   - `social/threads/<slug>.md` (X thread, 5–10 tweets)
2. Review and edit the package.
3. Commit and push → article is live in ~3 minutes.
4. Open X. Paste tweets one at a time from the thread file. ~3 minutes.
5. Open Instagram on your phone. Upload the 6 slides as a carousel. Paste caption. ~5 minutes.
6. (Optional) Paste the LinkedIn post.

**What you need to provide:** nothing new. Just the willingness to spend 10 minutes posting.

**Trade-off:** zero infra cost, no API approvals, you stay in control. Doesn't scale past ~1 post a day before it gets tedious.

This is also the *recommended starting point* even if you eventually want full automation. You'll learn what works on each platform before automating it.

### Phase 2 — Automate the X thread (1–2 days of work)

X is the easier side to automate. Add it next.

**What we'd build:**

- A GitHub Actions workflow that triggers when an article ships.
- The workflow reads `social/threads/<slug>.md`, posts each tweet as a thread.
- Logs the thread URL back as a PR comment (or a Slack ping if you'd rather).

**What you need to provide:**

1. **An X developer account.** Free tier works for thread posting at our volume. Sign up at developer.x.com.
2. **An X App** with OAuth 2.0 user-context auth.
3. **API credentials.** Specifically:
   - API key + API secret
   - Bearer token (for read; we mostly need write)
   - User access token + secret for the account you want to post from (this is the @jayporelabs account, presumably).
4. **Confirmation of the posting account.** I assume `@jayporelabs` but please confirm.

The credentials go into GitHub repo secrets — never in code. We use them like this in the workflow:

```yaml
env:
  X_API_KEY: ${{ secrets.X_API_KEY }}
  X_API_SECRET: ${{ secrets.X_API_SECRET }}
  X_ACCESS_TOKEN: ${{ secrets.X_ACCESS_TOKEN }}
  X_ACCESS_TOKEN_SECRET: ${{ secrets.X_ACCESS_TOKEN_SECRET }}
```

**Trade-off:** Free tier has rate limits (typically 50 writes / 24h on the free tier as of 2026). Plenty for a daily posting cadence. If you ever push higher volume you'd need to upgrade to Basic ($200/mo) or Pro.

### Phase 3 — Automate Instagram carousels (the hard one)

IG is the hardest of the three. Meta's official path requires:

1. **A Facebook Page.** Free to create. The Page is what owns the IG content publishing permission.
2. **An Instagram Business or Creator account.** If `@jayporelabs` is currently a Personal account, you need to convert it. Conversion is reversible.
3. **Linking IG to the Facebook Page.** Done in the IG app's settings.
4. **A Meta Developer App** with:
   - `instagram_basic`, `instagram_content_publish`, `pages_show_list`, `pages_read_engagement` permissions.
5. **App Review approval from Meta** for the `instagram_content_publish` permission. This is the friction point. Approval takes 1–4 weeks and Meta will ask for screencasts and a written use-case justification.
6. **A long-lived Page Access Token** with the above permissions.
7. **Publicly hosted images.** Meta's API can't accept local PNGs; it pulls images by HTTPS URL. We solve this by serving the carousel slides from `jayporelabs.com/social-assets/<slug>/slide-N.png` (Vercel can serve them; we just need to expose the `social/carousels/` directory or upload to a CDN).

**What you need to provide for Phase 3:**

1. The Facebook Page (existing or newly created).
2. The IG Business/Creator account (convert if currently Personal — keeps your followers, just changes the type).
3. The IG account linked to the Page (one-time setting in the IG app).
4. Time / patience for Meta's app review (this is the part you can't speed up).
5. Decision on the public-image hosting path:
   - **Option A** — expose `social/carousels/` via a public path on `jayporelabs.com`. Cheap, simple, but the images become publicly indexable.
   - **Option B** — upload to a CDN (Cloudflare R2, AWS S3 + CloudFront, etc.). More setup; cleaner separation.
   - I'd default to A unless you have a reason to keep slides private.

**Trade-off:** Real engineering work plus Meta's review queue. You're committing to the workflow for the long haul.

### Phase 3 alternative — third-party scheduler

If Phase 3's Meta-review process is too much friction, a third-party scheduler is the practical alternative:

- **Buffer.** Free tier supports IG carousel scheduling. Their API can be automated.
- **Later.** Similar feature set; their API is a bit nicer.
- **Typefully.** Excellent for X threads; less strong for IG.

You'd connect your IG account to the scheduler once (which uses the same Meta auth under the hood, but the scheduler has already done the App Review work). Our workflow would call the scheduler's API to post.

**Trade-off:** $15-50/mo per scheduler. Faster setup. You don't own the integration.

## What I recommend

**For the next 30 days:** Phase 1 only. Manual posting. You'll learn what works on each platform and what doesn't, and you'll have real data on what kinds of articles draw which audiences. Optimising automation before you have that data is premature.

**Day 31 onwards:** add Phase 2 (X automation). It's the highest-impact, lowest-friction win. Two days of my work; 10 minutes of credential collection on your side.

**When you're posting consistently and want to scale:** decide between Phase 3 (Meta direct) and Phase 3 alt (Buffer). The decision factor is volume and willingness to handle Meta's app review.
